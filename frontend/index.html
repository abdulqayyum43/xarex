Probe Setup Guide Step-by-step
1
Prerequisites
Confirm these are installed on the machine where the probe will run:
  • ✓ Linux, macOS, or WSL2 (Ubuntu 22.04+ recommended)
  • ✓ Go 1.21+ — verify: go version
  • ✓ Nmap — install: sudo apt install nmap -y
  • ✓ sudo / root access (required for raw socket scans)
  • ✓ Network access to the target subnet you want to scan
2
Download or build the probe binary
Download the pre-built binary from the Deploy Probe page, or build from source:
git clone https://github.com/your-org/xarex.git && cd xarex/probe go build -o xarex-probe . && echo "Build successful"
Pre-built binaries are available for Linux (x64/ARM) and Windows. Go to Deploy Probe for download links and Docker options.
4
Launch the probe
Connect to Xarex first to generate this command.
The probe registers with the Cloud Brain via gRPC (port 50051) and waits for scan tasks. Keep this terminal open while scanning.
5
Verify connection & launch a scan
Within 5–10 seconds a green online badge appears in the table above. Then go to Dashboard → Quick Scan and enter a target subnet (e.g. 192.168.1.0/24).
Troubleshooting
Probe not appearing?
Verify port 50051 (gRPC) is open between probe and Brain. Check firewall rules.
Permission denied errors?
Always run with sudo. Nmap SYN scan requires raw socket access.
Scan stuck at HOST_DISCOVERY?
Verify you can ping targets from the probe machine. Check subnet CIDR is correct.
Build fails (go: not found)?
Install Go: sudo snap install go --classic or visit golang.org/dl

Connect Xarex to your network in about 3 minutes — no security expertise required.

What's a probe — and why do I need one?

A probe is a tiny program (about 25 MB) you install on any computer inside the network you want to test — a laptop, a server, a Raspberry Pi, or a spare desktop. Once it's running it does one thing: it scans your network from the inside and reports back to Xarex.

Think of it like a security camera placed inside your office. The Xarex dashboard you're looking at is the monitor — it can't see what's inside your network until you put a camera there. The probe is that camera.

1 You run a single command on a computer inside your network.
2 The probe connects to xarex.com and waits for instructions.
3 You start a scan from this dashboard — findings stream back in real time.

Your connection details

The probe needs these three values to talk to Xarex. They're filled in for you automatically in the steps below — but here they are if you'd rather copy them by hand.

Before you start2-minute check
💻
A computer inside the network you want to scan. Any modern Linux server, Windows PC, or Mac will work — even a Raspberry Pi. A laptop you leave running during the scan is fine.
🔑
Admin (root) access on that computer. The probe needs to send low-level network packets, which requires admin privileges. On Linux that means sudo; on Windows, "Run as Administrator".
🌐
Outbound internet access to xarex.com. The probe makes outbound connections only — no firewall ports need to be opened on your side. If you can browse the web from that computer, you're set.
⏱️
About 3 minutes. Pick a method below (Linux is fastest), copy-paste the commands, and you'll see your probe appear in the dashboard.

Pick a method

All three options end up doing the same thing. Use whichever fits the computer you're installing on.

Linux — 3 steps, about 2 minutesRecommended
1
Download the probe
Open a terminal on the Linux machine inside your network and paste these commands. They download the probe and mark it as executable.
Terminal
curl -L https://xarex.com/download/xarex-probe-linux -o xarex-probe chmod +x xarex-probe
2
Save your settings to a file
This creates a small text file called xarex.conf next to the probe with your Org ID. The probe reads it automatically on startup — your real values are filled in below.
Terminal
cat > xarex.conf << EOF ORG_ID=loading… CLOUD_BRAIN_ADDR=xarex.com:50051 PROBE_ID=probe-$(hostname) EOF
3
Start the probe
Run this final command. You'll be asked for your password (that's sudo — it lets the probe send the network packets it needs). Leave the terminal window open while scanning; closing it stops the probe.
Terminal
sudo ./xarex-probe --config xarex.conf
✅You're done. Check the green banner near the top of this page — it should switch to "connected" within 30 seconds. If it doesn't, scroll to Troubleshooting below.
Docker — 1 commandIf you already use Docker
1
Start the probe container
One docker command does everything — pull the image, configure it, and start it in the background. The container restarts automatically if it crashes or the machine reboots.
Terminal
docker run -d \ --name xarex-probe \ --restart unless-stopped \ --network host \ --cap-add NET_RAW \ --cap-add NET_ADMIN \ -e ORG_ID=loading… \ -e CLOUD_BRAIN_ADDR=xarex.com:50051 \ -e PROBE_ID=probe-$(hostname) \ ghcr.io/xarex/probe:latest
💡--network host lets the probe see your network the way the host machine does. NET_RAW + NET_ADMIN are the minimum capabilities it needs to send scan packets.
2
Watch the probe come online
Tail the container logs to confirm it connected. You should see "probe registered" within a few seconds. Press Ctrl-C to stop watching — the probe keeps running in the background.
Terminal
docker logs -f xarex-probe
✅The banner at the top of this page should switch to "connected" within 30 seconds.
Windows — 3 steps in PowerShellRun as Administrator
⚠️Open PowerShell as Administrator: press the Start button, type "PowerShell", right-click it, choose "Run as administrator". The probe needs admin rights to send network packets.
1
Download the probe
This downloads the probe binary into your current folder.
PowerShell
Invoke-WebRequest -Uri "https://xarex.com/download/xarex-probe-windows.exe" -OutFile xarex-probe.exe
2
Save your settings to a file
Creates a small xarex.conf file next to the probe. Your real Org ID is filled in below.
PowerShell
@" ORG_ID=loading… CLOUD_BRAIN_ADDR=xarex.com:50051 PROBE_ID=probe-$env:COMPUTERNAME "@ | Out-File xarex.conf -Encoding UTF8
3
Start the probe
Leave the PowerShell window open while scanning — closing it stops the probe.

Troubleshooting

Most setup issues fall into one of these buckets. If you're still stuck, email support@xarex.com.

My probe isn't appearing in the dashboard.
Something's blocking the connection to xarex.com:50051. Test it from the probe machine: nc -zv xarex.com 50051. If that fails, your firewall is blocking outbound TCP on port 50051 — ask your IT team to allow it.
"Permission denied" or "raw socket" errors.
The probe needs admin rights to send the network packets it uses. On Linux, run it with sudo. On Windows, open PowerShell as Administrator. With Docker, make sure --cap-add NET_RAW is on your command.
My scan is stuck on "Host discovery".
The probe can't reach your targets. Try ping <target-ip> from the probe machine. Also check the subnet you entered is right (e.g. 192.168.1.0/24, not just 192.168.1.1).
My Docker container keeps crashing.
Check the error: docker logs xarex-probe. The most common cause is a wrong ORG_ID or a missing --network host flag. Re-copy the command from the Docker tab above — the Org ID is auto-filled.
TLS / certificate errors.
Your probe machine's system clock is probably off, or it can't reach the public certificate authorities. Confirm the time is right (date on Linux) and that the probe can reach https://xarex.com in a browser.
Web app / SSL findings missing.
Those scans need outbound HTTP and HTTPS access (ports 80 and 443) to your scan targets — not just ping. Check those ports aren't blocked between the probe and your targets.

Quick Start

Xarex is an autonomous penetration testing platform. A lightweight probe runs inside your network, connects to the Cloud Brain, and executes multi-phase scans that produce AI-driven security intelligence. Here's how to go from zero to your first scan.

Before You Begin

ℹ️You need a running Cloud Brain (the server) before deploying a probe. If you haven't set up the Cloud Brain yet, see the Deploy Probe page for Docker Compose setup instructions.

Step-by-Step

1

Connect to the Cloud Brain

Enter your Cloud Brain URL (e.g. http://your-server:8005) and your API key. Click Connect. A green status dot confirms the connection.

2

Deploy a Probe

Go to Deploy Probe in the sidebar. Copy your Org ID and Cloud Brain address. Download the binary for your OS and run it on a machine inside the target network. The probe appears in the Probes list within 30 seconds.

3

Launch Your First Scan

From the Dashboard, click Quick Scan. Enter a subnet (e.g. 192.168.1.0/24). Select your scan modules and click Launch. The live terminal shows probe output in real time.

4

Review Findings

When the scan completes, go to Findings to see all vulnerabilities sorted by severity. Click any finding for CVE details, CVSS score, MITRE mapping, and remediation steps.

5

Generate a Report

Go to Reports, select the scan, choose a template (Executive / Technical / Compliance), and click Generate Report. Export as PDF, HTML, or Markdown for sharing.

✅A typical first scan of a /24 subnet takes 3–8 minutes end-to-end. Critical findings appear in real time during the scan — you don't need to wait for completion.

Deploying Probes

The Xarex Probe is a statically compiled Go binary that runs inside your target network. It has no local dependencies, uses minimal resources (~30MB RAM), and communicates back to the Cloud Brain exclusively over encrypted gRPC (port 50051).

Architecture

🔒The probe never opens an inbound port. All communication is outbound only from probe to Cloud Brain. No port forwarding or inbound firewall rules are required on the probe host.

Deployment Methods

A

Linux Binary (Recommended)

Download the pre-built binary. Create xarex.conf with your Org ID and Cloud Brain address. Run with sudo ./xarex-probe --config xarex.conf. For production, install as a systemd service (see Deploy Probe → Persistent Installation).

B

Docker Container

Use docker run with --network host and --cap-add NET_RAW. Suitable for containerised environments. Use Docker Compose for persistent deployments with automatic restart.

C

Windows (PowerShell)

Download the .exe binary. Run as Administrator. Use NSSM to install as a Windows Service for persistence. Full PowerShell commands are in the Deploy Probe page.

Security Best Practices

  • Use AmbientCapabilities=CAP_NET_RAW in systemd instead of running as root
  • Store the API key in /etc/xarex/probe.conf with permissions 600
  • Only deploy the probe on hosts inside the network you have authorization to scan
  • Rotate the Org ID API key periodically via Settings → API Keys

Scan Modules

Xarex's scan engine is modular. You choose which modules to run per scan. Core modules run in every scan; extended modules add deeper coverage at the cost of scan time.

Core Modules (always enabled)

📡

Host Discovery

ICMP ping sweep + ARP broadcast to identify all live hosts in the target subnet. Falls back to TCP SYN probe if ICMP is blocked. Results feed all subsequent modules.

🔍

Port Scan

TCP SYN scan of the top 1,000 ports plus common service ports. Uses raw sockets for speed. Identifies open, closed, and filtered states. Configurable port ranges via Settings.

⚙️

Service Fingerprinting

Banner grabbing + protocol negotiation on every open port. Identifies service name, vendor, and version string (e.g. Apache httpd 2.4.51). This data drives CVE matching.

🎯

Vulnerability Scan

Matches discovered service versions against the NVD CVE database. Assigns CVSS v3.1 scores. Checks for known exploits (Metasploit, ExploitDB). Generates the bulk of findings.

Extended Modules

New in this release
🔒

SSL/TLS Audit

Connects to every HTTPS/TLS port and evaluates: certificate validity & expiry, cipher suite strength (flags SSLv3, TLS 1.0, RC4, etc.), HSTS, certificate transparency, and chain trust. Generates findings for weak configurations with remediation commands (nginx/Apache config snippets).

🌐

Web Application Recon

For every HTTP/HTTPS port: fingerprints web server, detects CMS (WordPress, Drupal, Joomla), checks security headers (CSP, X-Frame-Options, Referrer-Policy), scans common paths (/admin, /.env, /backup), and tests for directory listing. Non-destructive — read-only probing only.

🔑

Default Credentials Check

Tests a curated list of manufacturer default and commonly-used credentials against SSH, FTP, Telnet, HTTP Basic Auth, and common admin interfaces (router GUIs, network gear). Uses a safe, rate-limited approach to avoid account lockouts.

📂

SMB Enumeration

Connects to Windows/Samba shares on port 445. Enumerates shares, checks null session access, detects SMBv1 (EternalBlue risk), checks signing requirements, and lists share permissions. Critical for Active Directory environments.

🗂️

DNS Enumeration

Queries the target's DNS servers for zone information, SPF/DKIM/DMARC records, subdomains via bruteforce and dictionary, reverse lookups, and DNSSEC validation. Identifies misconfigurations that enable phishing or DNS hijacking.

🏢

Active Directory / LDAP Enumeration

Anonymously queries LDAP on port 389/636 to enumerate users, groups, computers, and Group Policy Objects. Checks for common misconfigurations: Kerberoastable accounts, AS-REP roastable users, unconstrained delegation, and password policy weaknesses.

⚠️Default Credentials and AD Enumeration modules may trigger IDS/IPS alerts. Only use against systems you have explicit written authorization to test.

Findings & Severity

Findings are vulnerabilities, misconfigurations, and security weaknesses discovered during a scan. Each finding is scored using CVSS v3.1 and enriched with attack context.

Severity Levels

CRITICALCVSS 9.0–10.0. Immediate exploitation risk. Patch within 24 hours. Likely includes remote code execution or full system compromise.
HIGHCVSS 7.0–8.9. Significant risk requiring prompt remediation within 72 hours. Often requires credentials or adjacent network access.
MEDIUMCVSS 4.0–6.9. Requires specific conditions to exploit. Schedule remediation within 30 days.
LOWCVSS 0.1–3.9. Low immediate risk. Address in next maintenance window. Can compound with other findings in attack chains.

Finding Fields

📋Each finding includes: CVE ID, CVSS v3.1 base score & vector, affected host and service, MITRE ATT&CK technique, attack path participation count, PoC availability flag, and AI-generated remediation steps.

False Positive Management

Open any finding and click Mark False Positive. The finding is suppressed from future reports and the risk score is recalculated. Suppressed findings are retained for audit trail purposes and can be reinstated at any time.

Attack Path Modeling

Attack Path Modeling shows you the specific sequence of vulnerabilities an attacker would chain to move from an initial entry point to a high-value target — the kill chain for your specific environment.

How It Works

1

Graph Construction

After the vulnerability scan, the probe builds a directed graph where nodes are hosts and edges are exploitable vulnerabilities (can host A compromise host B via this CVE?).

2

Path Enumeration

All paths from Internet-facing hosts to internal targets are enumerated. Each path has a composite risk score based on CVSS scores along the route and hop count.

3

Prioritisation

Paths are ranked by risk. The finding that appears in the most paths is your highest-priority fix — patching it collapses the most attack chains simultaneously.

💡Use the Network Map page to visualise attack paths as directional arrows between hosts. Click any arrow to see the specific vulnerability enabling that edge.

CVE Watch

CVE Watch continuously monitors the NIST National Vulnerability Database (NVD) and instantly flags when a new CVE matches any service version discovered in your environment.

How Matching Works

The Cloud Brain runs a background job every hour, fetching CVEs published in the configured time window (7, 14, or 30 days). It cross-references each CVE's CPE (Common Platform Enumeration) with your discovered service inventory. A match generates a CVE Watch alert with the "YOUR ENV" tag.

Key Indicators

YOUR ENVThis CVE matches a service version running in your environment. High priority.
PoCA public proof-of-concept exploit or Metasploit module exists. Treat as actively exploitable.
⏱️CVE Watch checks require at least one completed scan to have a service inventory to match against. Run an initial scan first, then CVE Watch will begin surfacing matches.
